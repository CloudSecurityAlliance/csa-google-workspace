# Code Audit — `csa-google-workspace`

**Date:** 2026-07-22
**Scope:** Comprehensive audit of the `csa-google-workspace` Python library — security, correctness, code quality/design, and test coverage.
**Version audited:** `0.1.0` (source: `src/csa_google_workspace/`, 1,308 LOC across 15 modules).
**Nature:** Findings-only review. **No source code was modified.** Recommendations are advisory.

---

## Executive summary

`csa-google-workspace` is a small, cleanly-layered wrapper over four Google Workspace APIs (Drive/Docs/Sheets/Slides). The architecture is genuinely coherent: a `Backend` `Protocol` seam with a real `ApiBackend` and an in-memory `FakeBackend`, a `Workspace` factory that MIME-sniffs and returns typed `Doc`/`Sheet`/`Slides`, dataclass domain models with `from_api()` normalizers, and centralized, idempotency-aware error translation. Both `mypy src` and `ruff check src` pass clean, and the offline unit suite is green (**169 passed, 9 skipped, 86.70 % branch coverage**).

The codebase shows deliberate, correct engineering in several security- and correctness-sensitive spots — `defusedxml` on attacker-influenceable XLSX comment text, `0o600` token writes into a `0o700` directory, a retry policy that never re-issues non-idempotent writes on 5xx, and correctly-paginated comment listing. There are **no critical or high-severity active defects** (no memory-safety-equivalent bugs, no injection, no SSRF, no secret exfiltration path).

The material findings cluster in four areas:

1. **OAuth privilege posture** — the library requests **full-Drive** scope and is **read-write by default**, giving every consenting user a maximally-privileged, long-lived token.
2. **One high-likelihood correctness bug** — spreadsheets with **numeric/year-named tabs** (e.g. `"2024"`) break `Sheet.as_text()` because the tab name is emitted as an unquoted A1 range.
3. **Test blind spots on security- and corruption-critical code** — the OAuth token lifecycle, `ApiBackend` pagination, and the `idempotent=False` write wiring are all only exercised by human-gated suites that never run in CI, so regressions in them ship green.
4. **A cluster of type/design rough edges** — an LSP-breaking `Sheet.create_comment` override, a `cast(Backend, None)` pattern that hides a real crash from the type checker, cross-document-type duplication, and bare `dict`/`list` hints on a `py.typed` public surface.

**Totals:** 3 High · 8 Medium · 12 Low · 6 Informational (29 findings).

## High-priority findings

### 1 — [High] Over-broad OAuth scope (full Drive) combined with read-write by default
**Location:** `auth.py:11-17`; defaults at `workspace.py:19,33,42`
**Category:** security

`_RW` requests `https://www.googleapis.com/auth/drive` — full read/write to **every** file in the user's Drive — plus `documents`, `spreadsheets`, `presentations`. A library whose job is reading/writing the specific documents a caller opens and managing their comments needs only the far narrower `drive.file` scope (access limited to files the app opens or creates). Compounding this, `read_only` defaults to `False` in `Workspace.__init__`, `from_credentials`, and `from_oauth`, so the default consent screen asks for full read-write to the entire Drive, and the persisted refresh token carries that authority indefinitely.

**Impact:** Every user who consents grants a long-lived token that can read, modify, and delete *any* file in their Drive — not just the documents handed to the library. If the token file (see #3, #17) or the embedding application is later compromised, the blast radius is the whole Drive rather than a handful of documents.

**Recommendation:** Prefer `drive.file` (or `drive.readonly`) unless full-Drive enumeration is genuinely required, and document the trade-off. Consider defaulting `read_only=True` with writes opt-in, or at minimum loudly documenting that the default acquires full-Drive write scope.

> *Severity note:* the security specialist rated this **Medium**; it is ranked #1 here because it is the default posture, the token is long-lived, and the blast radius is the entire Drive — a large, easily-reduced over-privilege that a security-focused consumer (CSA) will care about most.

### 3 — [High] OAuth credential lifecycle and token-file hardening are untested offline
**Location:** `auth.py:33-59`; `workspace.py:43-45`
**Category:** tests

`load_credentials` (the entire body) and `Workspace.from_oauth` have **zero** offline coverage — only the pure helpers `scopes_for`/`needs_reconsent` are tested. The token-cache branching (cached-token load, reconsent → re-auth, expired-with-refresh-token → `creds.refresh()`) and the security-sensitive persistence (`os.chmod(token_dir, 0o700)`, `os.open(..., 0o600)`) run only in `tests/oauth/`, gated behind `CSA_GW_OAUTH=1` **and** requiring a human at a browser + real Google — so it never runs in CI.

**Impact:** A regression that widens the token-file mode (dropping `0o600`/`0o700`) would leak a long-lived refresh token to other local users **with no failing test**. A logic error in the reconsent gate could silently reuse a token missing required scopes, or fail to refresh — surfacing only in production.

**Recommendation:** Add offline tests that monkeypatch `os.path.exists`, `Credentials.from_authorized_user_file`, `InstalledAppFlow.from_client_secrets_file`, and `google.auth.transport.requests.Request` to drive each branch: (a) valid cached token returned as-is; (b) insufficient-scope cached token → reconsent; (c) expired + refresh_token → `refresh()` called; (d) no token → flow invoked. Write to a `tmp_path` and **assert the resulting file mode is `0o600` and the dir `0o700`**, and that a `ValueError`/`GoogleAuthError` from `from_authorized_user_file` is wrapped as `AuthError`. None of this needs a browser.

### 6 — [High] `ApiBackend.list_comments` pagination has no offline test
**Location:** `backend.py:224-237`
**Category:** tests

The `nextPageToken` loop is the only non-trivial control flow in `ApiBackend` (the rest are one-line passthroughs) and is entirely uncovered offline. `test_apibackend_errors.py` stubs only `get_file_metadata`; `FakeBackend` returns all comments in one shot, so no offline test exercises pagination. Real coverage exists only in the never-in-CI integration suite.

**Impact:** A pagination bug — dropping the last page, an infinite loop when `nextPageToken` is echoed, or losing `startModifiedTime`/`includeDeleted` on the second request — ships undetected. On any document with **>100 comments** the library would silently return a truncated list.

**Recommendation:** Extend the `_Services`/`_Request` stub pattern already in `test_apibackend_errors.py` to a `.comments().list()` returning two pages (page 1 with `nextPageToken`, page 2 without). Assert the concatenated result is complete and that the second call carried `pageToken` and preserved `includeDeleted`/`startModifiedTime`.

## Medium-priority findings

### 2 — [Medium] Numeric / digit-leading tab names are emitted as unquoted A1 ranges
**Location:** `documents/sheet.py:26-30` (`_quote_tab`)
**Category:** correctness — *highest-likelihood active bug in the codebase*

The quoting heuristic is:

```python
if title.replace("_", "").isalnum() and not re.fullmatch(r"[A-Za-z]{1,3}\d+", title):
    return title           # unquoted
```

A purely numeric title (`"2024"`) or a digit-leading alnum title (`"1stQuarter"`) passes `.isalnum()` and does **not** match the cell-reference pattern (which requires leading letters), so it is returned **unquoted** and passed directly as the `range` argument to `values.get`. *(Confirmed by trace: `_quote_tab("2024") → "2024"`; `_quote_tab("café") → "café"` — non-ASCII names are also left unquoted, a lesser variant.)*

**Impact:** A spreadsheet with a year-named tab (`"2024"`, `"2025"` — extremely common) calls `get_values(range="2024")`. A bare number is not a valid A1 range → Sheets returns `400 Unable to parse range: 2024` → surfaces as `ApiError`. `Sheet.as_text()`, `as_text(tab="2024")`, and the `_cell_map` export path all fail for such a sheet.

**Recommendation:** Leave a title unquoted only if it matches `^[A-Za-z_][A-Za-z0-9_]*$` **and** is not a cell-ref pattern; quote everything else (all-digit, leading-digit, and non-ASCII). At minimum add `title.isdigit()` (and a leading-digit check) to the force-quote condition.

### 4 — [Medium] `os.chmod(0o700)` mutates a pre-existing, caller-supplied token directory
**Location:** `auth.py:52-55`
**Category:** security

```python
token_dir = os.path.dirname(token_path)
if token_dir:
    os.makedirs(token_dir, exist_ok=True)
    os.chmod(token_dir, 0o700)      # runs even if the dir already existed
```

The `chmod` runs unconditionally. If a caller points `token_path` at a file inside a pre-existing, possibly-shared directory (e.g. `/some/shared/dir/token.json`), the library forcibly changes that directory's mode to `0o700`.

**Impact:** A silent, surprising permission change on a directory the library doesn't own — can lock other users/processes out of a shared directory (availability), a side effect the caller never requested from a "load credentials" call.

**Recommendation:** Only `chmod` a directory the library just created — check `os.path.exists` before `makedirs`, or use `os.makedirs(mode=0o700)` and skip the `chmod` when the directory already existed. Hardening the token file itself (already done correctly) is sufficient in the pre-existing-dir case.

### 5 — [Medium] `Comment`/`Reply` inject dependencies via `cast(<T>, None)`, hiding a real crash from the type checker
**Location:** `comments.py:57-60`, `comments.py:98-100`
**Category:** quality

`_backend: Backend = field(default=cast("Backend", None), ...)` (and `_file_id: str`, `_comment_id: str`) tell mypy these are always populated, while `from_api()` leaves them `None` until `CommentCollection._wrap()` / `_attach_reply()` back-fill them. A `Comment`/`Reply` built through the public `from_api()` classmethod and then mutated (`.edit()`, `.reply()`, `.delete()`) calls `self._backend.<...>` on `None` → `AttributeError`, **with no mypy warning** because the field is declared `Backend`, not `Backend | None`.

**Impact:** The type system actively hides a crash path on a public constructor. `Comment` is exported; a consumer that reasonably calls `Comment.from_api(raw)` and mutates it gets a runtime `AttributeError` instead of a typed error or a static complaint.

**Recommendation:** Either make the injected fields `Backend | None` and guard mutators with a clear error ("detached Comment; obtain via `Workspace`"), or fold construction+injection into a single factory and stop exposing a half-initialized object.

### 7 — [Medium] Nothing verifies `ApiBackend` writes pass `idempotent=False`
**Location:** `backend.py:244-254,259-276,301-332`
**Category:** tests

`_errors.call`'s idempotent/non-idempotent retry behavior is well tested at the function level. But no test proves each `ApiBackend` write method (`create_comment`, `create_reply`, `update_*`, `delete_*`, `docs_batch_update`, `sheets_values_update/append/clear`, `*_batch_update`) actually passes `idempotent=False`. The code comment on `sheets_values_append` (`backend.py:313`) explicitly flags this as load-bearing ("a retried request would add the rows twice"), yet the wiring is unverified.

**Impact:** A regression dropping `idempotent=False` from any write would cause a 5xx-triggered retry to double-apply the mutation — duplicate appended rows, duplicate comments — the exact data-corruption failure the flag exists to prevent, with green tests throughout.

**Recommendation:** Inject a stub `_errors.call` (or a service whose `.execute` raises 503 once then succeeds) and assert each write does **not** retry on 5xx (underlying `.execute` called exactly once, `ApiError` propagates), or spy on the `idempotent` kwarg per method.

### 10 — [Medium] `Sheet.create_comment` breaks Liskov substitution
**Location:** `documents/sheet.py:61` vs `base.py:29`
**Category:** quality

`CommentsMixin.create_comment(self, content: str) -> Comment` is overridden by `Sheet.create_comment(self, text: str, cell: str | None = None)`. The first positional parameter is renamed `content` → `text`, an extra `cell` parameter is added, and the override drops the return-type annotation. `Doc`/`Slides` inherit the base `content=` keyword; `Sheet` does not. mypy misses it because the base lives on a mixin it sees as a separate class.

**Impact:** Code written generically against `Document` (`doc.create_comment(content="x")`) raises `TypeError` when the object is a `Sheet` — the keyword name of the same logical method depends on the concrete subtype.

**Recommendation:** Keep the parameter named `content` in the override and add `cell` after it; annotate `-> Comment`. Consider promoting the optional `cell` anchor into the base signature (ignored by Doc/Slides) so the contract is uniform.

### 11 — [Medium] `replace_text` / `batch_update` duplicated across document types
**Location:** `documents/doc.py:28-32,49-51`; `documents/slides.py:40-44,53-55`; `documents/sheet.py:123-126`
**Category:** quality

`Doc.replace_text` and `Slides.replace_text` are byte-for-byte identical, including the brittle reply-parse `(resp.get("replies") or [{}])[0].get("replaceAllText", {}).get("occurrencesChanged", 0)`. All three types repeat the `self._require_writable(); return self._backend.<x>_batch_update(...)` shape, and `Sheet` layers `self._cell_map_cache = None` by hand across five methods.

**Impact:** A fix to the fragile occurrences-parse must be made in two places and can drift; the Sheet cache-invalidation is a copy-paste a new mutating method can silently forget.

**Recommendation:** Extract a `parse_replace_all_text(resp) -> int` helper shared by both `replace_text` implementations; consider a base `_batch_update` template and a Sheet-level wrapper that invalidates the cache so individual methods don't each repeat it.

### 12 — [Medium] Bare `dict`/`list` and untyped `credentials` weaken the shipped type hints
**Location:** `backend.py:11-37` (Protocol); `workspace.py:33`; `documents/sheet.py:23,106,111,123`; `documents/doc.py:49`; `documents/slides.py:53`
**Category:** quality

The package ships `py.typed`, but the public interface is pervasively unparameterized: the `Backend` Protocol mixes `list[dict]` with bare `list`/`dict`; `Sheet.values() -> list`, `update(..., values: list)`, `batch_update(requests: list) -> dict`; and `Workspace.from_credentials(cls, credentials, ...)` gives `credentials` no annotation (inferred `Any`) — the single most important public entry point has an untyped parameter.

**Impact:** Downstream mypy/pyright users (the reason `py.typed` exists) get `Any` for cell values, request bodies, and credentials — no autocomplete, no misuse detection. `check_untyped_defs` verifies bodies internally but does nothing for callers.

**Recommendation:** Parameterize consistently — `list[list[Any]]` for cell values, `list[dict[str, Any]]` for request lists, `dict[str, Any]` for responses, and `credentials: google.auth.credentials.Credentials`. A `JsonDict = dict[str, Any]` alias would make the Protocol uniform.

### 14 — [Medium] FakeBackend can diverge from ApiBackend with no test to catch it
**Location:** `backend.py:40-188` (FakeBackend) vs `191-333` (ApiBackend)
**Category:** tests

FakeBackend raises typed exceptions directly rather than routing raw `HttpError` through `translate_http_error`, does not paginate, and hard-codes "MEASURED" behaviors — flipping parent `resolved` on a resolve reply (`:106`) and stripping `content`/`author` on delete (`:126-129`). These are asserted offline, but only the gated integration suite proves the *real* API matches them. No conformance test asserts both classes satisfy the same `Backend` Protocol surface.

**Impact:** The offline suite can stay green while ApiBackend drifts — a Protocol method implemented only on FakeBackend, or Google changing the resolve/delete semantics the fake hard-codes. "Green tests, broken prod."

**Recommendation:** Add a Protocol-conformance test asserting every `Backend` method exists with a matching `inspect.signature` on both backends. Document each "MEASURED" behavior with a pointer to the integration test that validated it.

### 15 — [Medium] Error-translation edge cases untested
**Location:** `_errors.py:20-21,33-34,45,48-49`
**Category:** tests

Every error-body-parse helper has an untested failure branch: `_reason_and_message` on a malformed/undecodable body (`20-21`), `_service_disabled_details` on a malformed body (`33-34`), and `_retry_after` on a non-integer header (`48-49`) or dict-resp missing the key (`45`). All existing tests feed well-formed JSON and clean headers.

**Impact:** A real Google 4xx/5xx with an HTML error page, empty body, or an HTTP-date `Retry-After` (valid per spec) hits untested paths; if any raised instead of degrading, the translator would mask the original error with a confusing `KeyError`/`ValueError`. (See also the *active* negative-`Retry-After` bug, #8.)

**Recommendation:** Feed `translate_http_error`/`call` an `HttpError` with (a) non-JSON `content` (`b"<html>503</html>"`), (b) `Retry-After: "Mon, 01 Jan 2030 00:00:00 GMT"` and `""`, and assert graceful degradation (no exception escapes the translator).

## Low-priority findings

### 8 — [Low] Negative/malformed `Retry-After` → uncaught `ValueError` from `time.sleep`
**Location:** `_errors.py:38-49` (`_retry_after`), `_errors.py:85` (sleep in `call`)
**Category:** correctness — *active bug*

`_retry_after` does `int(value)` with no lower bound; `Retry-After: -5` parses to `-5`. In `call`, `_sleep(min(retry_after, 60) ...)` → `min(-5, 60) = -5` → `time.sleep(-5)` raises `ValueError`. Only `HttpError` is caught in the loop, so the `ValueError` propagates uncaught.

**Impact:** A misbehaving or hostile server returning a negative `Retry-After` on a 429 turns a retryable rate-limit into an uncaught `ValueError`, breaking the "callers never touch raw errors" contract.

**Recommendation:** Clamp: `max(0, min(retry_after, 60))`, or return `None` from `_retry_after` when `value <= 0`.

### 9 — [Low] 401 Unauthorized is misclassified as `ApiError`, never `AuthError`
**Location:** `_errors.py:52-68` (`translate_http_error`)
**Category:** correctness

The translator branches on 404, 403 (incl. `SERVICE_DISABLED`), and 429, then falls through to `ApiError`. There is no `401` branch, though the hierarchy defines `AuthError` for "bad/expired creds, consent needed". *(Confirmed: `translate_http_error(<401>) → ApiError`.)*

**Impact:** A live 401 (token revoked/expired mid-session, or scope surfaced as 401) reaches callers as `ApiError(status=401)`; code catching `AuthError` to trigger re-consent will not fire. Mitigated by `google-auth` auto-refreshing most expiries before the HTTP call.

**Recommendation:** Add `if status == 401: return exc.AuthError(message or "authentication failed")` ahead of the fallthrough.

### 13 — [Low] `read_only=True` is enforced only in application code, not at the token scope
**Location:** `auth.py:20-29` (`needs_reconsent`); guards at `base.py:54`, `comments.py:116`
**Category:** security

When a caller requests `read_only=True` but a cached token already carries the read-write base scope, `needs_reconsent` treats the RW scope as satisfying the RO requirement (`auth.py:25-27`) and reuses it. Read-only-ness is then enforced purely by `ReadOnlyError` guards in Python.

**Impact:** Defense-in-depth gap — a code path that bypasses `_require_writable`/`_guard`, or direct use of the returned `Credentials`, could still mutate Drive despite the user asking for read-only. The OAuth layer provides no backstop.

**Recommendation:** Document that `read_only` is an app-layer restriction, not a scope restriction. Optionally acquire/use a genuinely read-only token when `read_only=True` rather than reusing a broader cached one.

### 16 — [Low] A run with both insertion and deletion suggestions silently drops the deletion
**Location:** `suggestions.py:24-31` (`_collect`)
**Category:** correctness

`sid = (ins or dele)[0]; kind = "insertion" if ins else "deletion"`. When a run has both `suggestedInsertionIds` and `suggestedDeletionIds`, it is unconditionally classified as an insertion under the insertion id; the deletion id is discarded. *(Confirmed: `ins=["i1"], dele=["d1"]` yields only `Suggestion(id="i1", kind="insertion")`.)*

**Impact:** In documents where a suggested insertion is itself suggested for deletion (or with overlapping suggestions), `Doc.suggestions` under-reports — a real pending deletion never appears.

**Recommendation:** Emit an entry per distinct id present on the run, or document that overlapping suggestions collapse to the insertion.

### 17 — [Low] Token file opened without `O_NOFOLLOW` (symlink follow / TOCTOU)
**Location:** `auth.py:56`
**Category:** security

`os.open(token_path, os.O_WRONLY | os.O_CREAT | os.O_TRUNC, 0o600)` follows a symlink if `token_path` already exists as one, and `O_TRUNC` truncates the target. The `0o600` mode only applies on *creation*, so an attacker-planted symlink or a pre-existing loose-mode file is written/truncated with its existing perms.

**Impact:** If an attacker can create files in the token directory, they could redirect the token write or induce truncation of an arbitrary user-writable file. Largely mitigated by the `0o700` dir in the user's home by default → low exploitability.

**Recommendation:** Add `os.O_NOFOLLOW`; consider verifying the path is not a symlink before writing, and re-`chmod` to `0o600` after truncating a pre-existing file.

### 18 — [Low] XLSX export parsing is not bounded against a decompression bomb
**Location:** `_cellmap.py:36-60` (`parse_xlsx_comments`); consumed by `documents/sheet.py:82`
**Category:** security

`zipfile.ZipFile(io.BytesIO(xlsx_bytes))` then `z.read(name)` on every persons/threadedComments entry, with no cap on decompressed size or entry count. `defusedxml` blocks XML entity expansion but does **not** bound zip decompression, and the comment text inside is attacker-influenceable (per the module's own docstring).

**Impact:** A crafted spreadsheet with highly-compressible comment XML could inflate to a large in-memory string on `z.read`, causing memory pressure / DoS in the calling process. Bounded to a self/second-party DoS since the archive is Google's export of the user's own or shared-to-them document.

**Recommendation:** Check `ZipInfo.file_size` before `read`, cap the entry count, and/or read with a bounded byte limit; reject archives whose declared uncompressed size exceeds a threshold.

### 19 — [Low] Underlying auth-library error string interpolated into `AuthError`
**Location:** `auth.py:40,49`
**Category:** security

`raise AuthError(f"could not load or refresh cached credentials: {e}") from e` embeds the string form of the underlying `ValueError`/`GoogleAuthError`. In practice these surface `invalid_grant`-style codes rather than token values, so direct secret leakage is unlikely — but the message is not guaranteed secret-free and may land in logs less protected than the token file.

**Recommendation:** Emit a fixed message and rely on the `from e` chained traceback for developer detail, so sensitive content is not baked into the string that tends to get logged.

### 20 — [Low] Dead `Location.tab` field; `row=0/col=0` silent sentinel
**Location:** `comments.py:16-20`; `_cellmap.py:28,33`
**Category:** quality

`Location` declares `tab: str | None = None`, but the only constructor (`location_from_ref`) never passes it — it is always `None`. That same function returns `row=0, col=0` on an unparseable ref while still setting `cell=ref`.

**Impact:** An exported field that looks meaningful is permanently `None`, misleading users who branch on `comment.location.tab`; the `0/0` sentinel is silently wrong for a bad ref.

**Recommendation:** Populate `tab` from the worksheet the comment XML belongs to, or drop the field until implemented. Document the `0/0` sentinel or raise instead.

### 21 — [Low] Fragile string surgery and magic constants in Sheet/Doc
**Location:** `documents/sheet.py:67` (`self.url.split('/edit')[0]`), `documents/sheet.py:50` (`"A1:Z1000"`), `documents/doc.py:42` (`endIndex", 2 ... end - 1`)
**Category:** quality

`create_comment` builds a cell deep-link by splitting `webViewLink` on the literal `/edit`; if the URL lacks it, the whole URL is used and the anchor is silently malformed. `as_text` falls back to a hard-coded `A1:Z1000` window that silently truncates larger grids (contradicting the method's own "not silently truncated" docstring for the metadata-present path). `Doc.append_text` uses bare `2`/`end - 1` arithmetic with no explaining constant.

**Recommendation:** Build the deep link from `self.id` rather than parsing `webViewLink`; name the fallback range a documented constant (or page through); name the Docs index constants.

### 23 — [Low] cellmap/sheet fallback branches untested
**Location:** `_cellmap.py:28` (`location_from_ref` fallback); `documents/sheet.py:36,38` (`_gid` no-match), `documents/sheet.py:50` (`as_text` no-metadata)
**Category:** tests

The malformed-ref fallback (`row=0,col=0`), `_gid` returning `0` for a nonexistent tab (→ wrong deep-link target in `create_comment(cell=...)`), and the `as_text` `A1:Z1000` fallback are all uncovered.

**Recommendation:** Parametrize `location_from_ref` over `""`, `None`, and a two-letter col (`"AB3"` → `col == 28`); add a fixture with named tabs to pin `_gid("NoSuchTab")` behavior (0 vs raise); open a Sheet whose `get_spreadsheet` returns `{"sheets": []}` and assert the fallback render.

### 24 — [Low] Function-local `import re` on a per-render hot path
**Location:** `documents/sheet.py:27` (`_quote_tab`)
**Category:** quality

`import re` sits inside `_quote_tab`, called once per tab on every `as_text()`. `re` is module-level in `comments.py` and `_cellmap.py` — this is inconsistent and adds minor per-call overhead. (The genuinely-gated imports in `Sheet._cell_map` — `zipfile`, `defusedxml` — are defensible where they are.)

**Recommendation:** Hoist `import re` to the top of `sheet.py`.

### 29 — [Low] `filter(author=...)` exclusion and email-match arms untested
**Location:** `comments.py:206-208`
**Category:** tests

Only the matching case of the author filter is tested; the exclusion branch and the `email`-match arm (`c.author.email == author`) are never asserted. This is behavior users rely on — a bug excluding matching comments could silently drop results.

**Recommendation:** Add a `filter(author=...)` test asserting a non-matching author is excluded and that matching by `email` works.

## Informational

### 22 — [Info] Cell→location matching is a whole-second + exact-content heuristic
**Location:** `_cellmap.py:19-22` (`_second`), `:63-74` (`match_locations`)
**Category:** correctness

Matching keys on `(author, content, whole-second timestamp)`. Sub-second timestamps are truncated, so instants reported as `…59.9xx` (Drive) vs `…00.0xx` (XLSX) straddle a second boundary and fail to match; any content-normalization difference (HTML vs plain, or the deep-link suffix `Sheet.create_comment` appends) also defeats equality. By design ("no confident match ⇒ `location=None`"), but it means legitimately-anchored comments can silently come back with `location=None`.

**Recommendation:** Consider a ±1s timestamp window and/or content normalization before comparison; document the limitation. *(Not a regression — design-level.)*

### 25 — [Info] `Workspace.open_by_url` is fully redundant with `open`
**Location:** `workspace.py:29-30` — `open_by_url(url)` just calls `self.open(url)`, and `open` already runs `parse_file_id`, which accepts a URL or a bare id. Two public names for one operation. Drop it, or mark it an explicit alias in the docstring.

### 26 — [Info] Load-bearing types are not exported
**Location:** `__init__.py:9-13` — `.comments` returns a `CommentCollection` and `Workspace.__init__` takes a `Backend`, yet neither (nor `ApiBackend`/`FakeBackend`/`Document`) is in `__all__`. Users type-annotating these intermediate values must import from private-looking submodules. Export `CommentCollection` and `Document`; optionally `Backend`. (That `FakeBackend` ships in the runtime package rather than a test-support module is a minor smell.)

### 27 — [Info] `Comment`/`Author` `repr()` exposes document content and author email
**Location:** `comments.py:29-41` (`Author`), `83-100` (`Comment`) — the internal `_backend`/`_file_id`/`_comment_id` fields are correctly `repr=False`, but `content`, `html_content`, `quoted_text`, and `Author.email` (PII) are in the default dataclass `repr`. Logging a `Comment`/`Author` (or an exception carrying one) renders that data. This is the data the caller intentionally handles — flagged so downstream consumers log these objects deliberately. If PII-in-logs is a concern, mark those fields `repr=False` or provide a redacting `__repr__`.

### 28 — [Info] Runtime dependencies pinned with lower bounds only
**Location:** `pyproject.toml` — all four deps use `>=` with no upper bound. This is normal and appropriate for a library (avoids downstream resolver conflicts), and `defusedxml>=0.7` is a safe floor. No action for the library; embedding applications should pin/lock transitively and run a dependency scanner.

## Verified strengths

These held up under review and are worth preserving:

- **`_errors.call()` retry policy** — always retries 429, retries 5xx only when `idempotent=True`, caps `Retry-After` at 60 s, and injects `_sleep` for testability. Correct and clearly reasoned. (Function-level tests are thorough; the gap is the per-method *wiring*, #7.)
- **Security hygiene done right** — `defusedxml` on attacker-influenceable XLSX comment text with a documented rationale; token written via `os.open(..., 0o600)` into a `0o700` dir. (The only stdlib `xml.etree` import, `documents/sheet.py:73`, is used solely as a `ParseError` exception type — it never parses untrusted input.)
- **`ApiBackend.list_comments` pagination is correctly implemented** — the loop terminates on absent `nextPageToken` and preserves filters (the finding, #6, is the *absence of a test*, not a bug).
- **Sheets writes default to `value_input_option="RAW"`**, which neutralizes formula/CSV injection (values stored literally, not interpreted).
- **Domain models document each Drive-API quirk they normalize** ("absent ⇒ False (MEASURED)"), and `pyproject.toml` explains every non-obvious choice (the `E702` ignore, the coverage threshold, the static-version AST parse, the missing-stub overrides).
- **`mypy src` and `ruff check src` both pass clean**; `FakeBackend` uses `copy.deepcopy` throughout to prevent test-fixture aliasing.

## Methodology & limitations

**How this audit was produced.** Four independent specialist review passes — security, correctness/bug-hunt, code quality/design, and test coverage — each read the full `src/` tree and relevant tests, and ran the toolchain where applicable. Every reported finding was then **cross-checked against an independent full read** of all 15 source modules; file:line references and the two traced bugs (numeric-tab quoting, negative `Retry-After`) were verified before inclusion. Anticipated false positives (e.g. "stdlib `xml` is used to parse untrusted input") were checked and excluded.

**What was run:**
- `pytest -q --cov=csa_google_workspace --cov-branch` → **169 passed, 9 skipped, 86.70 % branch coverage** (offline `FakeBackend` suite).
- `mypy src` → *Success: no issues found in 16 source files*.
- `ruff check src` → *All checks passed!*

**Limitations:**
- The **`tests/integration/` and `tests/oauth/` suites were not run** — they require real Google credentials + a browser and are gated behind `CSA_GW_INTEGRATION=1` / `CSA_GW_OAUTH=1`. `ApiBackend`'s real-Google behavior and the OAuth flow were reviewed **statically only**. Findings #3, #6, #7, and #14 are precisely about the fact that this untested surface is where security- and corruption-critical logic lives.
- Severities are **advisory** and reflect a blended priority (severity × likelihood × blast radius); the per-finding "Severity note" callouts flag where the ranking departs from a specialist's raw label.
- No dynamic/fuzz testing of the XLSX/XML parsing path was performed (finding #18 is reasoned, not exploited).
- This is a **findings-only** review — no code was changed.
