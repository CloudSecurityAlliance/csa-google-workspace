# Code Audit — `csa-google-workspace`

**Date:** 2026-07-22
**Version audited:** `0.1.0` (`src/csa_google_workspace/`, 1,308 LOC, 15 modules)
**Nature:** Findings-only review. **No source code was modified.** All recommendations are advisory.

**Deployment assumption (the lens for this whole audit).** The library is evaluated against its stated trajectory, not as a generic package:

1. an **interactive MCP server** that lets a user review a document's comments with an AI, reply to and resolve them, and edit the document; and
2. an **autonomous AI/API automation** — e.g. a backend that sweeps a user's documents for open comments and helps resolve them.

Both act **on a user's behalf**. The current code is an explicit **proof-of-concept** (hence the local `token.json` / installed-app flow); the target credential path is server-managed OAuth handing the library ready credentials via `Workspace.from_credentials`.

---

## Executive summary

`csa-google-workspace` is a small, cleanly-layered, and **unusually self-aware** PoC. The architecture is right for where it's going: a `Backend` `Protocol` seam with real/fake implementations, a `Workspace` factory returning typed `Doc`/`Sheet`/`Slides`, dataclass domain models that normalize documented Drive-API quirks, and centralized idempotency-aware error translation. `mypy src` and `ruff check src` pass clean; the offline suite is green (**169 passed, 9 skipped, 86.70 % branch coverage**); and `TODO.md` already scopes an "audit → cleanup → PyPI" milestone, tracks an `examples/` MCP server and an open async decision, and explicitly defers threat-model calls "to the security audit."

**So the distance between this PoC and the target system is not primarily code flaws.** It is three things, in priority order:

- **A. Two capability boundaries** the target workflows will hit (suggestions are read-only via the API; the library has no document *discovery*).
- **B. Operational contract that hasn't caught up to "on a user's behalf"** — which credential seam is production vs interim, the concurrency/async rule, and the confused-deputy security posture. The prose vision is correct; the *deployment contract* is unwritten.
- **C. A short list of genuine, net-new code bugs** — most importantly a spreadsheet tab-name quoting bug that breaks `as_text()` on year-named tabs, plus a few error-handling and typing issues that bite specifically in the on-behalf-of/automation context.

There are **no critical/high-severity active security defects** — no injection, no SSRF, no secret-exfiltration path — and several deliberately-correct security choices (see *Verified strengths*).

This report is organized by that structure (A → B → C), followed by **D. the threat-model decisions the audit recommends** (the ones `TODO.md` is waiting on), a reconciliation with the team's existing backlog, and methodology/limitations.

## A. Fitness for the two target workflows

### A1 — Interactive comment-review MCP server

**Well-supported for the comment loop.** The workflow "list open comments → understand each in context → reply / resolve → edit the doc" maps cleanly:

| Step | API | Status |
|------|-----|--------|
| List open comments | `doc.comments.filter(resolved=False)` | ✅ supported |
| Understand a comment in context | `Comment.quoted_text` (anchored snippet), `.content`, `.replies`, `.author` | ✅ the model captures `quotedFileContent` — exactly the context an AI needs |
| Reply / resolve / reopen / edit | `Comment.reply()`, `.resolve()`, `.reopen()`, `.edit()` | ✅ full comment mutation |
| Edit the document | `Doc.replace_text`, `insert_text`, `append_text`, `delete_range`, `batch_update` | ⚠️ present, but see caveat |

**Caveat — doc edits are index-based.** `insert_text(text, at: int)` and `delete_range(start, end)` require the caller to compute character offsets into the Docs linear index. An AI computing those offsets is an error-prone pattern (off-by-one against the document's implicit indices). **`replace_text(find, replace)` is the safe, intent-level primitive** — MCP tool design should steer models toward it and expose raw index edits only as an advanced escape hatch.

**Boundary — "process them" stops at comments.** If the review workflow includes acting on **suggestions** (suggesting-mode edits), that is **not possible via the API**: `Doc.suggestions` reads them (and `as_text(suggestions="accepted"/"rejected")` previews the outcome), but accept/reject raises `UnsupportedOperation` — Google exposes no endpoint; it needs the unbuilt `PlaywrightBackend`. Net: **comments — full workflow; suggestions — read/preview only.** This should be stated as a first-class limitation of the MCP server, not discovered at runtime.

### A2 — Autonomous document-sweep automation

**The per-document half is a strong fit.**

- **Incremental sweeps work:** `comments.filter(since=…)` plumbs through to Drive's `startModifiedTime`, so a recurring job can pull only comments changed since its last run.
- **Safe under retry:** every write passes `idempotent=False`, so a 5xx-triggered retry won't double-post a reply or double-resolve — the correct default for an unattended job.
- **Complete results at scale:** `ApiBackend.list_comments` correctly paginates `nextPageToken`, so a document with >100 comments isn't silently truncated.

**Boundary — the library does not discover documents.** There is **no file listing/search** (`ApiBackend` exposes `get_file_metadata` for one id, but no `files.list`/query). The library is **document-scoped**: you hand it an id/URL and it operates on that file. "Sweep a user's documents" therefore requires the *automation* to enumerate which documents to sweep — via the Drive `files.list` API directly — and then open each with `Workspace.open(id)`. This is a legitimate scope decision, but it is the first wall use-case #2 hits and it is not stated anywhere. Recommend documenting the division: **the library owns per-document comment/content work; discovery is the caller's job** (with a short `files.list` snippet in the README).

**Error-handling readiness for long-running jobs.** Three code findings from §C bite hardest *here* and should be prioritized for the automation use case specifically:
- **401 is not mapped to `AuthError` (C-bug #9)** — a long-running job whose token dies mid-run gets an opaque `ApiError`, so it can't cleanly catch `AuthError` to trigger re-auth.
- **A negative/malformed `Retry-After` crashes the retry loop (C-bug #8)** — a hostile/misbehaving 429 turns a retryable rate-limit into an uncaught `ValueError`.
- **Pagination is untested (C-bug #6)** — the exact silent-truncation risk a sweep depends on not having.

## B. Documentation & contract coherence with "on a user's behalf"

The vision prose is already in the README/spec. What's missing is the **operational contract** for running the library on someone else's behalf — the things that only bite at deployment.

### B1 — Label the interim vs. target credential story
The README shows `from_oauth("client_secret.json")` / `token.json` as an ordinary usage path and says "**Writes are on by default.**" For the target model that framing is misleading. The docs should state plainly:

> The installed-app flow (`from_oauth`, local `token.json`) is **PoC/CLI scaffolding**. In production, the MCP server (or automation host) runs the OAuth **web** flow itself, stores per-user tokens in its own secret store, and passes the library ready credentials via **`Workspace.from_credentials(creds)`** — the production seam. `from_oauth`'s `run_local_server()` cannot run on a server.

This single reframing resolves most of the token-file findings in §C by relocating them to the interim path (see §C2).

### B2 — Write down the concurrency / async contract
`TODO.md` lists "**Async story — decide**" as open; it is on the critical path for *both* workflows and should be decided as part of this audit. Verified facts to base the contract on:

- **State is per-instance — there is no global credential singleton.** `ServiceRegistry`, `ApiBackend`, and `Workspace` each hold their own credentials/clients; `Sheet._cell_map_cache` is per-instance. So **per-request/per-user `Workspace` instances are cleanly isolated** — a real strength for multi-tenant use, and worth stating as a guarantee.
- **`googleapiclient` service objects are not thread-safe.** So the contract is: *one `Workspace` per request/user; do not share a `Workspace` (or its backend) across threads.*
- **The stack is synchronous.** Recommended (matches TODO's lean): **document the `asyncio.to_thread` pattern** rather than build an async facade.
- **Token refresh is the embedder's job, and the library correctly doesn't fight it.** On the `from_credentials` path the library never persists refreshed tokens — google-auth refreshes the in-memory access token per request; durable storage belongs to the server. State this so embedders don't expect persistence.

### B3 — Document the on-behalf-of security model
An MCP server or automation wielding a user's full-Drive token is a textbook **confused-deputy** setup. The library can't solve that (only the server can ensure it does exactly what the user asked), but it should document the primitives it provides and the posture it expects:

- **Read-only as the default posture for embedders** (see §D2) — the library already maps `read_only=True` to read-only *scopes* on a fresh `from_oauth` consent, and credentials are per-`Workspace`, so a server can hold a read-only `Workspace` and a separate read-write one and route each tool call to the narrower.
- **PII in logs (C-finding #27).** `Comment`/`Author.repr()` renders document `content`/`quoted_text` and author `email`. For chatty multi-tenant servers "acting on behalf of users," that puts other users' document text and emails in shared logs — a compliance concern worth an explicit warning (or a redacting `__repr__`).
- **Per-user credential isolation** — document that a `Workspace` must never be reused across end users (it binds one user's credentials).

Detail on the ones most worth reading:

#### C-#2 — [Medium] Numeric / digit-leading tab names break `as_text()` *(highest-likelihood active bug)*
`documents/sheet.py:26-30` — `_quote_tab` leaves a title unquoted when `title.replace("_","").isalnum()` and it doesn't match the cell-ref pattern `[A-Za-z]{1,3}\d+`. A year-named tab (`"2024"`) is all-digits → `.isalnum()` is `True`, and the cell-ref pattern requires *leading letters* so it doesn't match → **returned unquoted** → `get_values(range="2024")` → Sheets `400 Unable to parse range: 2024`. `as_text()`, `as_text(tab="2024")`, and the `_cell_map` export path all fail for that sheet. *(Verified by trace: `_quote_tab("2024") → "2024"`; non-ASCII names like `"café"` are also left unquoted — a lesser variant.)* **Fix:** leave a title unquoted only if it matches `^[A-Za-z_][A-Za-z0-9_]*$` and is not a cell-ref; quote everything else (all-digit, leading-digit, non-ASCII).

#### C-#9 — [Medium] 401 never mapped to `AuthError`
`_errors.py:52-68` branches on 404/403/429 then falls through to `ApiError`; there is no `401` branch, though `AuthError` exists for "bad/expired creds." An on-behalf-of server that catches `AuthError` to trigger re-consent won't fire on a mid-session token death. **Fix:** `if status == 401: return exc.AuthError(message or "authentication failed")` ahead of the fallthrough. *(Mitigated in practice by google-auth auto-refreshing before most calls.)*

#### C-#8 — [Low] Negative `Retry-After` crashes the retry loop
`_retry_after` (`_errors.py:38-49`) does `int(value)` with no floor; `Retry-After: -5` → `time.sleep(min(-5,60)) = time.sleep(-5)` → `ValueError`, and only `HttpError` is caught in `call` (`:85`) so it escapes untyped. **Fix:** `max(0, min(retry_after, 60))`, or return `None` for non-positive values.

#### C-#5 — [Medium] `cast(Backend, None)` hides a crash from the type checker
`comments.py:57-60,98-100` declare injected fields as non-`Optional` via `cast(<T>, None)`, so a `Comment`/`Reply` built through the public `from_api()` and then mutated calls `self._backend.<...>` on `None` → `AttributeError`, with **no mypy warning**. `Comment` is exported. **Fix:** make the fields `Backend | None` with a guarded "detached object" error, or fold construction+injection into one factory.

#### C-#12 — [Medium] Untyped `credentials` on the production entry point
`workspace.py:33` — `from_credentials(cls, credentials, ...)` gives `credentials` no annotation (inferred `Any`). This was minor when it was one path among several; under the target model it is **the** production entry point for every MCP/automation embedder, on a `py.typed` package. **Fix:** annotate `credentials: google.auth.credentials.Credentials`; parameterize the `Backend` Protocol's bare `dict`/`list` (a `JsonDict = dict[str, Any]` alias helps).

*(Full detail for the remaining C1 rows — #10, #11, #14, #15, #16, #18, #21, #24, #26, #27, #29 — carries over verbatim from the finding descriptions; each is one focused fix. #18 is re-rated Low→Medium because in a shared multi-tenant server one user's malicious sheet can degrade the process for all tenants, not just themselves.)*

## D. Threat-model decisions the audit recommends

`TODO.md` parks several calls "for the security audit." Here they are.

1. **OAuth scope — keep full `drive`; document the breadth.** It's required for the product. Do **not** narrow to `drive.file`. Add a consent-breadth note to the README: authorizing this reads, edits, and deletes **anything** in the user's Drive.
2. **Read-only as the embedder default.** Recommend MCP/automation hosts instantiate a `read_only=True` `Workspace` and escalate to a write-capable one deliberately, per operation. It already maps to read-only *scopes* on fresh `from_oauth` consent; on the `from_credentials` path the embedder controls scope, so the guidance is "acquire read-only creds unless the tool genuinely writes."
3. **Credentials in CI (the deferred question).** Do **not** store long-lived full-Drive user creds in CI. Prefer PyPI **Trusted Publishing (OIDC)** for release (already TODO's preferred path), and for any live-suite CI job use a **dedicated, disposable, read-only test account** via manual `workflow_dispatch`, never a real user's token.
4. **Async — decide now: document, don't build.** Ship the `asyncio.to_thread` pattern in the README (§B2). An async facade over a sync stack is high-cost, low-value.

## Verified strengths

Held up under an independent full read; worth preserving:

- **Idempotency-aware retry** (`_errors.call`) — always retries 429, retries 5xx only when `idempotent=True`, caps `Retry-After` at 60 s, injects `_sleep` for testability. Correct and clearly reasoned (the gap is per-method *wiring* tests, C-#7).
- **Correct pagination** in `ApiBackend.list_comments` (C-#6 is the missing *test*, not a bug).
- **Security hygiene done right** — `defusedxml` on attacker-influenceable XLSX comment text with a documented rationale; token written `os.open(..., 0o600)` into a `0o700` dir; the only stdlib `xml.etree` import (`sheet.py:73`) is used *solely* as a `ParseError` type, never to parse untrusted input; Sheets writes default to `RAW`, neutralizing formula/CSV injection.
- **Clean multi-tenant isolation** — no global credential state; per-request `Workspace` instances don't share auth.
- **Honest, empirical documentation** — models annotate each Drive-API quirk they normalize ("absent ⇒ False (MEASURED)"), `research/`+`experiments/` record probe-verified behavior, and `pyproject.toml`/`TODO.md` explain every non-obvious choice. `mypy` + `ruff` both clean.

## Methodology, prior review, and limitations

**Method.** Four independent specialist passes (security · correctness · code-quality · test-coverage), each reading the full `src/` tree and tests and running the toolchain; every reported finding was then **cross-checked against an independent full read** of all 15 modules before inclusion. File:line refs and the two traced bugs (numeric-tab quoting, negative `Retry-After`) were verified. Anticipated false positives (e.g. "stdlib `xml` parses untrusted input") were checked and excluded. This revision re-frames the same evidence against the stated deployment trajectory.

**Ran:** `pytest -q --cov --cov-branch` → **169 passed, 9 skipped, 86.70 % branch coverage**; `mypy src` → clean; `ruff check src` → clean.

**Relationship to the 2026-07-21 external review.** That review (TODO Tier 0) found + fixed the raw-`HttpError` leak and the silent cell-map degrade. This audit does not re-bill those; its net-new contribution is §A/§B and the C1 bugs listed above.

**Limitations.**
- **`tests/integration/` and `tests/oauth/` were not run** (need real Google creds + a browser). `ApiBackend`'s real-Google behavior and the OAuth flow were reviewed **statically only** — which is precisely why the untested surface (C-#3/#6/#7) is flagged.
- Severities are **advisory**, blended against the deployment assumption; the zip-bomb finding (#18) is reasoned, not exploited.
- No dynamic/fuzz testing of the XLSX/XML path.
- **Findings-only** — no code was changed.
